================================================================
QueryCraft — Test Module: Bugs to Fix & Improvements for Students
================================================================
Audience: Engineering team
Scope: Classic Test (/tests/[id]/attempt) and Interactive Quiz (/interactive-quiz/[id]/attempt)

Legend:
  [BUG]   = Defect that needs fixing
  [UX]    = User-experience improvement
  [A11Y]  = Accessibility
  [PERF]  = Performance / scalability
  [SEC]   = Anti-cheat / integrity refinement
  [REL]   = Reliability / resilience


----------------------------------------------------------------
1) BUGS / KNOWN ISSUES
----------------------------------------------------------------

1.1 [BUG] Submit confirmation uses native window.confirm()
    Where: src/app/(dashboard)/tests/[id]/attempt/page.tsx:733
    Problem: Native window.confirm() is jarring, can be auto-dismissed by
    some browsers, and gives no preview of unanswered questions.
    Fix: Replace with a styled modal that shows:
       - "X of Y questions answered"
       - List of unanswered question numbers (clickable to jump back)
       - Time remaining
       - Confirm / Cancel buttons

1.2 [BUG] No browser-navigation guard during an active attempt
    Where: attempt page has no `beforeunload` handler.
    Problem: Student can accidentally close the tab, hit Back, or refresh
    and lose context. Auto-save helps, but there is no warning.
    Fix: Add a `beforeunload` listener while attempt is in progress to
    warn before navigation. (Already needed for both classic + interactive.)

1.3 [BUG] Result page still shows "No submitted answer" for old attempts
    Root cause was fixed (LEFT JOIN question_bank for catalogue questions),
    but historical attempts created before the fix have no rows in
    attempt_answers because saveAttemptAnswers silently dropped them.
    Fix: Add a one-off migration / admin tool that lets a teacher
    re-run evaluation for already-submitted catalogue-based attempts,
    OR clearly mark such attempts as "data unavailable" in the UI.

1.4 [BUG] Auto-save fails silently on network errors
    Where: persistAnswers catch block in attempt/page.tsx
    Problem: If the network is down, the user sees no indication. Their
    answer remains in memory but is never persisted. On submit, the
    PATCH/POST may fail and answers can be lost.
    Fix:
       - Show a persistent "Offline — answers not saved" banner when
         a save fails (e.g. retry status 'error').
       - Queue failed saves in IndexedDB / localStorage and retry on
         reconnect.

1.5 [BUG] Hardcoded 30-min fallback timer
    Where: attempt/page.tsx:185 (`useState(30 * 60)`)
    Problem: If `test.duration_minutes` is missing, students get 30 min
    by default — could be wrong for short or long tests.
    Fix: Show a loading skeleton until duration is known, OR clamp to
    the test's actual duration on every tick rather than initial state.

1.6 [BUG] Interactive quiz: skipped questions don't always award 0
    Where: interactive-quiz/[id]/attempt/page.tsx — timeout handler
    Behaviour: When a question times out, the answer is recorded with
    `timedOut: true` but the timing/score logic should be audited to
    confirm the student receives a 0-point row (so they appear in the
    leaderboard with the correct totals).

1.7 [BUG] Tab-switch popup auto-submits on second offence with no clarity
    Where: attempt/page.tsx tab-switch handler
    Problem: First switch shows a warning. Second switch immediately
    auto-submits — students sometimes report they only switched briefly
    (e.g. notification popup). No grace period.
    Fix:
       - Configurable threshold per test (e.g. 3 strikes).
       - Distinguish between "tab hidden < 2s" (likely a notification)
         vs "tab hidden > 5s".
       - Show a 5-second countdown before forced submit so the student
         can return to the tab.

2) UX IMPROVEMENTS — CLASSIC TEST
----------------------------------------------------------------

2.1 [UX] Pre-test instructions screen
    Currently students land directly on Question 1. Add an intro panel
    showing:
       - Test name, duration, number of questions
       - Anti-cheat rules in plain language (no copy/paste, no tab
         switching, etc.)
       - "I understand and accept" checkbox + "Start Test" button
       - A timer that does NOT start until the student clicks Start.

2.2 [UX] Flag a question for review
    Add a "Flag" button next to each question. Flagged questions
    appear in a different colour in the question palette (the small
    numbered buttons at the top) so students can return to them.

2.3 [UX] Question palette visual states
    Currently the palette shows answered (highlighted) vs not. Extend to:
       - Not visited (grey)
       - Visited but unanswered (yellow)
       - Answered (green)
       - Flagged (orange ring)
       - Current (teal)

2.4 [UX] Final review screen before submit
    Before the actual submit POST, show a single "Review Answers" page
    with a list of questions, the student's selected option / typed
    answer, and a Submit button at the bottom.

2.5 [UX] Timer warnings
    Where: timer effect in attempt/page.tsx
    Add subtle visual + (optional) audio cues:
       - 5 minutes remaining → orange tint + toast
       - 1 minute remaining → red tint + flashing
       - 10 seconds → countdown beep (configurable, off by default)

2.6 [UX] Better SQL editor for sql_fill questions
    Currently a plain `<textarea>`. Replace with CodeMirror 6 or Monaco
    with:
       - SQL syntax highlighting
       - Bracket matching
       - Keyword auto-complete (SELECT, FROM, WHERE…)
       - Optional "Run query" sandbox to test syntax against a sample
         schema (won't validate correctness; only syntax).

2.7 [UX] Saved indicator already added — extend to show last-saved time
    e.g. "Saved 3s ago" so students trust the system.

2.8 [UX] Show student name + photo at the top of the attempt page
    Reinforces that the student is taking the test under their own
    account, reduces account-mix-up incidents.

2.9 [UX] Keyboard shortcuts
       - Arrow keys: previous / next question
       - 1-9: select MCQ option
       - Ctrl+Enter: submit (with confirmation)
       - F: flag for review

2.10 [UX] Auto-advance toggle for MCQs
     Optional setting: "Move to next question after selecting an answer".
     Speeds up confident students; off by default for cautious ones.

3) UX IMPROVEMENTS — INTERACTIVE QUIZ
----------------------------------------------------------------

3.1 [UX] Show points awarded after each question
    Already have feedback; ensure the points-earned animation includes
    a streak counter ("3 correct in a row! +50 bonus") to gamify.

3.2 [UX] Show position trend on leaderboard
    "You moved from #7 → #3" to keep students engaged between rounds.

3.3 [UX] Practice round
    Optional: 1-question warm-up before the actual quiz starts so
    students learn the UI without losing real points.

3.4 [UX] Audio feedback (opt-in)
    Short, configurable sounds for correct / wrong / time-up.

3.5 [UX] Visible per-question timer ring
    A circular progress ring around the question number gives a more
    intuitive feel of time pressure than a plain number.


----------------------------------------------------------------
4) ANTI-CHEAT REFINEMENTS
----------------------------------------------------------------

4.1 [SEC] Allow paste INTO the SQL textarea (not from outside)
    Currently all paste is blocked, even paste from earlier in the same
    field. This frustrates legitimate students who want to rearrange
    their query. Consider allowing same-document paste only, or a
    teacher-configurable toggle.

4.2 [SEC] Per-test anti-cheat policy
    The `anti_cheat_policy` column already exists. Surface it in the
    teacher UI:
       - "Strict" (block everything, force submit on tab switch)
       - "Moderate" (warn on tab switch, allow paste)
       - "Open" (no restrictions — practice tests, take-home)

4.3 [SEC] Fullscreen enforcement (optional)
    For high-stakes tests, request fullscreen on Start; auto-submit if
    the student exits fullscreen. Configurable per test.

4.4 [SEC] Webcam/microphone proctoring (opt-in, future)
    Out of scope today, but the violation_logs schema can already
    accommodate richer event types.

4.5 [SEC] IP / device fingerprint per attempt
    Log device fingerprint on attempt creation. If a second device
    posts answers for the same attempt_id, flag it (don't auto-block,
    but make it visible to the teacher in the review page).

4.6 [SEC] Question randomisation per student
    Already supported by interactive quiz settings. Add the same option
    for classic tests (shuffle question order using a seed derived from
    student_id + test_id) so adjacent students don't see the same
    sequence.


----------------------------------------------------------------
5) RELIABILITY / RESILIENCE
----------------------------------------------------------------

5.1 [REL] Local cache of answers (IndexedDB)
    Mirror `answers` to IndexedDB on every change so a browser crash
    doesn't lose work. On reload, offer to restore.

5.2 [REL] Service worker for offline question access
    Pre-cache the question payload and static assets so a flaky
    connection doesn't blank the page.

5.3 [REL] Submit retry with exponential backoff
    The submit POST currently fails hard on a single network blip.
    Retry up to 5 times with 1 / 2 / 4 / 8 / 16-second backoff before
    surfacing an error.

5.4 [REL] Idempotent submit endpoint
    Add an `Idempotency-Key` header (e.g. `attemptId + 'submit'`) so
    that a double-click or retry never creates duplicate evaluations.

5.5 [REL] Heartbeat ping
    Have the attempt page POST `/attempts/:id/ping` every 30s so the
    server can detect abandoned attempts and the teacher can see
    "active vs idle" students live.


----------------------------------------------------------------
6) ACCESSIBILITY (A11Y)
----------------------------------------------------------------

6.1 [A11Y] ARIA labels on all interactive controls
    Question palette buttons, option buttons, navigation buttons all
    need accessible names + role/state attributes (aria-current,
    aria-pressed).

6.2 [A11Y] Keyboard-only navigation
    Verify the entire attempt flow can be completed with only the
    keyboard. Currently Submit and option selection rely on click.

6.3 [A11Y] Screen-reader announcements
    Live region for:
       - Auto-save status ("Saved")
       - Timer warnings ("5 minutes remaining")
       - Tab-switch warning popup

6.4 [A11Y] Colour-contrast audit
    The teal/cyan accent on dark backgrounds is striking but may fail
    WCAG AA for some text sizes. Run an axe audit.

6.5 [A11Y] Reduced-motion support
    Respect prefers-reduced-motion for the leaderboard / point pop
    animations in the interactive quiz.

6.6 [A11Y] Increase font-size option
    Expose a +/- font-size toggle for visually impaired students.


----------------------------------------------------------------
7) MOBILE / SMALL-SCREEN
----------------------------------------------------------------

7.1 [UX] Sticky timer on mobile
    Currently the timer can scroll off-screen. Pin it to the top of
    the viewport on screens < 768px.

7.2 [UX] Bigger touch targets
    Option buttons and palette buttons should be at least 44 × 44 px.

7.3 [UX] Bottom-fixed action bar
    Previous / Next / Submit pinned to the bottom on mobile so they
    are always one tap away.

7.4 [UX] Avoid the SQL textarea on small screens
    Or at least default to a larger textarea height; current min-h is
    too short on mobile.


----------------------------------------------------------------
8) PERFORMANCE / SCALABILITY
----------------------------------------------------------------

8.1 [PERF] Batch auto-save with diff
    Already partly done — only changed answers are sent. Consider
    debouncing further (e.g. 1.5s) if many students cause API load.

8.2 [PERF] Connection pool tuning for high concurrency
    Default TEST_DB_POOL_MAX=30. For 100+ concurrent students raise
    to 80 via env var; document this in the deploy guide.

8.3 [PERF] Cache test + question fetch
    The result page fetches test, questions, and attempt every load.
    Add HTTP cache headers (private, short max-age) for the test +
    questions endpoints once the test is published.

8.4 [PERF] Avoid N+1 in evaluation
    Verify evaluateAttemptAnswers does not issue per-question queries.

8.5 [PERF] Add rate limiting on hot endpoints
    Protect /attempts, /submit, /interactive/check from abuse.
    Mentioned in docs/Roadmap.md but not yet implemented.


----------------------------------------------------------------
9) RESULT PAGE IMPROVEMENTS
----------------------------------------------------------------

9.1 [UX] Score breakdown by topic / difficulty
    If question_bank entries have tags or difficulty, group results so
    a student sees "70% in JOINs, 40% in subqueries".

9.2 [UX] Time-spent per question
    Already capturable from interactive quiz timing data. Add a small
    histogram.

9.3 [UX] Explanation per question (if teacher provides one)
    Add an `explanation` field to the question schema; show it on the
    result page beneath the correct answer when published.

9.4 [UX] Print / PDF download
    A "Download my result" button to save a PDF for records.

9.5 [UX] Share / retake
    For practice tests, allow a "Retake" button that creates a new
    attempt_number row.


----------------------------------------------------------------
10) TEACHER-FACING IMPROVEMENTS THAT BENEFIT STUDENTS
----------------------------------------------------------------

10.1 Teacher can preview the student attempt experience
     (read-only) so they can verify timing, anti-cheat severity, and
     question wording before publishing.

10.2 Teacher gets a "Likely confusing question" report — questions
     where < 30% of students answered correctly. Helps the teacher
     improve the question bank, which improves future student
     experiences.

10.3 Allow teacher to extend an individual student's time after the
     attempt has started (accessibility accommodation).

10.4 Teacher can reset / re-open a submitted attempt for a student
     who had a legitimate technical issue.
